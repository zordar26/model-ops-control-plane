# RFC 0001 — MVP Scope and Principles

**Status:** Draft  
**Owners:** Zordar  
**Last updated:** 2026-02-24

## Problem Statement

AI platform teams running multiple LLM providers (OpenRouter, Anthropic, Azure OpenAI, self-hosted Llama) lack a unified control plane. Today they stitch together:

- Prompt files spread across repos or notebooks
- Ad-hoc scripts for routing traffic and collecting logs
- Manual spreadsheets for cost and latency tracking
- Human-only quality gates before shipping prompt updates

This slows iteration, hides spend, and makes compliance reviews painful. We need an opinionated workflow that keeps experiments fast but production safe.

## MVP Goals

Deliver a CLI-first experience that:

1. **Describes routing + governance** in a single `route.yaml` + `evals/*.yaml` format.
2. **Ingests provider logs** (JSONL) and produces cost/latency/error summaries.
3. **Runs offline evals** before rollouts and blocks when regression thresholds fail.
4. **Outputs human-friendly reports** (terminal + Markdown) for stakeholders.

## Out of Scope (for MVP)

- Real-time traffic shifting or live proxying.
- UI dashboard (CLI + generated Markdown only).
- Vendor-specific key management.
- Fine-grained RBAC (single-team usage assumed).

## Personas

| Persona | Needs |
| --- | --- |
| AI Platform Lead | Compare providers, control spend, prove compliance |
| Applied ML Engineer | Update prompts/agents safely, get fast feedback |
| Compliance/Legal | Audit history of prompt changes and eval evidence |

## User Journeys

1. **Route definition**
   - Declare primary provider, fallbacks, and per-request guardrails in `route.yaml`.
   - Commit to git and review like infrastructure-as-code.

2. **Log analysis**
   - Drop JSONL traces (OpenRouter-compatible) into `logs/`.
   - Run `mops report --window 24h` to surface spend/latency/error metrics.

3. **Change validation**
   - Modify prompts or routes.
   - Execute `mops check` to run eval suites and generate a Markdown summary.
   - If thresholds fail, exit non-zero (ideal for CI).

4. **Stakeholder handoff**
   - Use `mops report` output as attachment in Slack/email to prove readiness.

## Functional Requirements

- Parse YAML configs with schema validation (Pydantic models).
- Read newline-delimited JSON logs with required fields (`timestamp`, `provider`, `model`, `prompt_tokens`, `completion_tokens`, `latency_ms`, `cost_usd`, `status`).
- Support configurable windows (`--window 24h`, `--since 2026-02-24T00:00Z`).
- Provide colorized terminal output (Rich) plus optional `--output reports/report.md`.
- Emit Markdown summaries suitable for Slack/email hand-offs.
- Make CLI composable via Typer subcommands.

## Non-Functional Requirements

- Python 3.10+ runtime.
-  <1s latency for 10k log lines on M3 laptop.
- 100% static typing coverage on core modules.
- Unit tests for log parser, cost aggregation, and CLI commands.

## Open Questions

1. Should we bundle connectors for proprietary log APIs or require users to export JSONL first?
2. Where do we persist historical reports (git vs. blob storage)?
3. How many evaluation metrics should be first-class vs. plugin-based?
